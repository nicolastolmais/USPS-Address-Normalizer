Identify Actors

1. Who will be using the system?
1.1 Non-technical person and technical person.

2. Will it be used by the public, one individual, or many users?
2.1 One individual at a time.

2.a Do we need to consider users with disabilites?
2.a1 Not at this time.

3. Will the user have a technical background?

4. Will the application be working with any other systems?
4.1 Yes, one mock-up database, but not Google Maps.

5. What are the specifications on the external database?
5.1 Just be given access to mock-up database that has address fields.

6. What entity or organization are we working for?
6.1 Statewide system housed at ATSU that maintains info on physicians, medical students, medical programs, and it has a lot of bad address data in it. If they connect home district, they get funds from legislature. Thus, half of the addresses will be from MO.

Scenarios

7. What should the system do with caught address errors that are normalizable?
7.1 The address should just be normalized with no user interaction.

7.aa What should the system do with addresses that are not normalizable?
7.aa1 In the case we detect but cannot normalize errors, we can have it go to a separate bad output file where the user can choose a variety of possible addresses or to operator.

7.a Should we fix spelling errors?
7.a1 Yes if it is safe to do so.

7.b Once we get normalized addresses, will there be an option to put them back in the database so we don’t have to renormalize them again?
7.b1 Might do this in the future, but not our problem at the moment.

7.c If the system crashes in the middle, what should happen?
7.c1 Restart the whole data. (Do whatever is the safest to not lose data).

7.d What should the system do with duplicate addresses?
7.d1 The output should notify the user about the duplication and which one it is. This should happen before and after normalization.

7.e Should we expect non-ASCII characters?
7.e1 Dr. Beck will get back to us about this.

Use Cases

8. How will this system be run?
8.1 Two ways this will be run. The first way is a non-technical person at computer with a program running where a person can type address and hit validate, and it will be validated. The user can then copy and paste the normalized address into the database. The second way is with a tech-savvy person who will provide batches of addresses of arbitrary size and wants it processed in a batch. Depending upon the types of errors found will depend on how things will preceed from there. The second option might include a flash drive and searching for correct folders also.

9. What should the system do with mixed up address fields?
9.1 Fields will not be mixed up. You can blindly assume that the first field is the address. More than one item may be stuff inside a field box though for Puerto Rico, so safety checks are needed.

9.a What specific checks can we do to make sure the data is right?
9.a1 Check state, city, zip-5, and zip-4 from static accesses to a database once a month. If zip-4 is not generated, try to automatically generate it.Might spell check (distance between this string and that string) and fix “closest one”. Might be a pattern in errors and figure out errors by that approach.

10. What should the system do if there are Spanish inputs or abbreviations?
10.1 The system should handle addresses appropriately.

Functional Requirements

11. Will there be a user interface or just a back-end system?
11.1 Ability to use user-interface and command line will be beneficial, and there will also be a back-end system.

12. What exactly are the inputs and outputs?
12.1 The inputs will be either one address or many addresses in all sorts of formats and mispellings. The outputs will be a new file with the addresses that were successful, ones that were not successful, and possibility of user interaction to choose the appropriate address. The output file should be configurable via command or set-up option. The output format will be agreed upon by the class and have the same fields as the input. This needs to be explored more for what exactly displays in the output file.

13. Should the output addresses be sorted or categorized? If so, how?
13.1 The outputs will be in the same order as the input, unless another sorting makes sense. Might have options for the user to choose how they want the output to be categorized.

13.a Can errors be deferred to the operator and have operator choose option?
13.a1 Yes. Options can be "option A", "option B", or "reject all". Do not let operator handle impossible errors, just kick it immediately to output file of issues.

14. What defines a valid input?
14.1 Post office standards. Will be a spot for address field, city, state, zip-5, and zip-4 with no names attached.

15. What invalid inputs are the system not responsible for handling?
15.1 If we can’t detect error, we can’t react to it (like wrong street number)

16. What are the most critical functions within the application?
16.1 The number one priority is to normalize correct addresses. The second priority is to recognize incorrect addresses so it does not get automatically accepted. Rather have questionable result rejected rather than making a guess and have it be wrong. (Do no harm). (Rather false negative, rather than false positive).Do not need to know if the address does exist, but if it could exist.

17. Will we be dealing with postal addressing standards, business addressing standards, or both standards?
17.1 Both types of addresses. Question from Client: will we need to know which addresses are for personal or business?

18. Will the inputs be supplied in plain text or in a structured file such as XML or CSV?
18.1 Inputs will probably be supplied in CSV formats.
